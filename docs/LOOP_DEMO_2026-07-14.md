# The optimizer/critic loop, demonstrated — and what it proves

Claude optimizes. Gemini critiques. Two runs, same loop, same models, same day. The only
difference was **how the task was scoped** — and that difference was everything.

---

## Run A — the badly-scoped task (task 1 from the catalogue)

> "Add Pydantic response models to **every** FastAPI endpoint in `src/interface/api.py`"
> — a **232 KB** file.

| | |
|---|---|
| Wall clock | **~5 hours** |
| Iterations | 3 (+ retries) |
| Score | **0.0** |
| Useful output | **none** |

Every evaluation timed out — `Gemini CLI timed out after 900s`, on every attempt **and**
every retry. The 0.0 was never a judgement: **the judge never ran.**

The optimizer was not failing. It was doing exactly what the task asked — producing 34 KB
candidates — and a CLI evaluator's latency is **super-linear** in prompt size (measured on
this repo: 5KB→7s, 20KB→17s, 50KB→**142s**). At that size the judge simply cannot answer.

Read as a scoreboard, this run says *"the model produced garbage."* That reading is false,
and it is exactly the misreading that makes months of work look like failure.

## Run B — the same loop, a well-scoped task

> "Add a `--json` flag to `scripts/verify_system.py` so CI can consume the gate results.
> Keep the human output as the default. Do not change any gate logic."

| | |
|---|---|
| Wall clock | **minutes** |
| Iterations | **1** |
| Score | **10.0 — converged** |
| Useful output | working code, **now in `scripts/verify_system.py`** |

---

## But the score is not the evidence. I ran it.

A 10/10 from a critic is an *opinion*. SAGE's own lesson is that opinions hallucinate, so the
candidate was put through the evidence gate:

```
$ python scripts/verify_system.py --fast          # human mode unchanged (default)
7/7 gates passed

$ python scripts/verify_system.py --fast --json   # new flag
{"gates": [{"name": "sidecar boots + handshakes", "passed": true, "detail": ""},
           {"name": "desktop RPCs answer (53/53)", "passed": true, "detail": ""}, ...],
 "passed": 7, "total": 7}
```

Valid JSON, human output preserved, no gate logic touched. **The 10/10 was correct** — but it
is only *known* to be correct because it was executed, not because a model said so.

---

## Two false alarms — both mine

While testing the candidate I twice "found bugs" that did not exist:

1. **"It crashes with `NotADirectoryError`."** It did — because I had run it from
   `docs/proposals/`, and the script derives `ROOT` from `__file__.parent.parent`. Wrong
   depth, wrong root, invalid subprocess cwd. **My harness, not its code.**
2. **"`--json` prints nothing."** It printed fine. I had piped stderr to `/dev/null` and then
   read the exit code of the *pipeline* instead of the process.

This is the third and fourth time in one session that **a broken instrument manufactured a
defect** — after the truncated JSON that made Gemini report "malformed output", and the
timed-out critic recorded as a score of 0.

**That is the real lesson, and it is not about which model you use.**

---

## What changed as a result

- **Oversize guard.** The evaluator now refuses a prompt it cannot judge (default 40 KB) and
  says *why*, instead of silently burning hours to arrive at a phantom 0.0. It aborts the
  loop rather than iterating — an oversize candidate does not shrink by retrying.
- **The message is explicit**: *"This is NOT a quality score — the judge never ran. Re-scope
  the task to a smaller unit of work."*

## The rule this establishes

> **Scope the task to something a judge can actually judge.**
> A model cannot rescue a task that cannot be evaluated — and an unevaluated task will report
> as a failure of the model.

The loop works. It converged in one iteration and produced code that passes an evidence gate.
It was never the model. It was the scope, and the instruments.
