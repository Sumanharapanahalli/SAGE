## 1. The instrument that was missing: `make verify`

Every unit suite in this repo passed green on a tree where **the app could not start**.
Unit tests import modules; nothing ever opened the product. That is how the launcher broke
without anyone noticing.

`make verify` answers one question — *can a person use SAGE right now?* — by exercising the
real artifacts: boot the sidecar → call every RPC the UI calls → compile the Tauri shell →
build the frontend → assert the launcher is shell-agnostic → assert the critic panel has
more than one voice.

It caught a bug **I** had introduced and committed, within a minute of existing. That is the
point of it.

**Run this before believing anything works.** `make test` is not that check.

## 3. The critic panel is real now (Claude + Gemini + local Qwen)

It was never a panel. `config.yaml` set `gemini_timeout: 30` as a "fast-fail", but Gemini's
measured latency on this repo is **5KB→7s, 20KB→17s, 50KB→142s**, and critic prompts
routinely exceed 20KB. Gemini timed out, silently left the pool, and **one judge wore the
costume of three**.

- `gemini_timeout` → 300s.
- A **local Qwen critic** (`qwen2.5-coder:7b`, via Ollama) now auto-registers as a third,
  cross-vendor voice — probed against the Ollama API first, so a critic that cannot answer is
  never *counted* as a critic. Being free, it can grade every candidate, which is what makes
  an always-on panel affordable.

Verified live: `gemini 35 | claude 42 | qwen 85 → median 42`. Qwen is markedly lenient — the
robust median is exactly what stops one lenient or hallucinating critic from moving the verdict.

## 4. Agents get their tools from Docker — nothing on the laptop

SAGE had **no MCP client**. `mcp_registry` imported in-process Python modules and called them
as functions: no server, no sandbox, every dependency installed on the host. (`.mcp.json` is
read by Claude Code, never by SAGE — and still points at `/home/shetty/...`.)

`src/integrations/mcp_docker.py` is a real MCP client. Each server runs as a container
(`docker run --rm -i mcp/fetch`) speaking MCP over stdio via the **official `mcp` SDK**, with
images from **Docker's official MCP catalog**. No protocol was reinvented.

- **network: none by DEFAULT.** A tool with ambient network access is an exfiltration path;
  only `fetch`, whose whole job is the network, opts in.
- **Mounts are read-only by default.** A tool that can write to the working tree can bypass
  the HITL gate by editing code directly.
- Tools register into the existing registry as `<server>.<tool>`, so agents, the ReAct loop
  and the desktop `mcp.tools` RPC all see them unchanged.

**Verified through the agent's own path:**
`mcp_registry.invoke("fetch.fetch", {...})` → executed in a container → returned live content.

## 5. Bugs found by *running* things (all fixed, all tested)

| Bug | Why it mattered |
|---|---|
| `_PROBE_POOL` undefined in `health.py` | Every LLM health probe raised `NameError` — the Status page could never report the LLM healthy |
| `make desktop-dev` used POSIX `VAR=x cmd` + cargo not on PATH | **The app would not start from PowerShell at all**, while 3265 unit tests stayed green |
| `open(path, "w")` with no encoding (6 runners) | LLM output containing `→` raised `UnicodeEncodeError` and **destroyed entire gym training sessions** |
| Constitution `voice` / `constraints` never validated | Bad data saved cleanly, then crashed `build_prompt_preamble()` — which is injected into **every agent's system prompt** |
| Failed critic scored as **0** | A critic that *could not answer* was recorded as one that judged the work worthless. With two failures the median collapses to 0, failing work rated 70 |
| 7 endpoint tests crashed on FastAPI | A crashing test cannot detect a missing endpoint — **false safety** |
| No way to unload/remove a solution | Operator could open a solution but never close one |

**Recurring theme, three separate times today: a failed instrument was mistaken for a bad
result.** That — not model capability — is what "months of drift" looks like from the inside.

## 6. What is NOT done (no pretending)

- **SAGE Evolve is a design, not code.** The two-tier optimizer/critic system
  (`model_panel`, `evidence`, `project_decomposer`, `evolve`) is specced
  (`docs/superpowers/specs/2026-07-13-sage-evolve-design.md`, debated with Gemini) and
  **not built**. It is days of work, not hours. Building it badly would just add to the pile
  of half-finished features that caused this mess.
- **pose-engine has not been touched.** I have never opened it. Importing an arbitrary repo is
  Level 0 of the Evolve spec and does not exist yet.
- **Eight handlers remain written-but-unregistered** — `logs`, `agentrun`, `regulatory`,
  `queuedb`, `knowledgesync`, `activity`, `safety`, `mr`. Dead code at the dispatcher. The
  Console/logs page in particular would crash if routed (5 pre-existing `tsc` errors are this).
- **`cargo test` cannot run here** — `STATUS_ENTRYPOINT_NOT_FOUND` from the WebView2 loader,
  reproduces on a clean tree. The Rust compiles; `unload_solution`'s runtime respawn is
  compile-checked only, not observed.
- **The 24 Gemini feature reviews are hypotheses, not a work order.** Two of the findings I
  hand-checked were false.
