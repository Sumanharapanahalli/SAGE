# SAGE Audit — 2026-07-13

Autonomous audit run while the operator was away. Everything below was **executed**, not
inferred from code reading. Commands and outputs are reproducible.

---

## Headline

**SAGE is in far better shape than "it isn't usable" suggests.** The desktop backend —
the whole app, since there is no HTTP layer — answers on every RPC the UI calls, and the
core SAGE loop (surface → propose → human approval gate → audit record) works end to end.

Three real bugs were found and fixed. The most consequential discovery is not a bug in
SAGE at all: **the tooling that measures SAGE has been lying**, in two separate ways.

---

## 1. Gemini restored as the critic (the original request)

The Evaluator-Optimizer loop had **Gemini removed from the critic seat**. Not from the core
module (`src/core/evaluator_optimizer.py` still documented "Claude optimizes, Gemini
evaluates") but from `scripts/self_improve.py` — the runner actually invoked. Its docstring
read *"Claude as BOTH optimizer and evaluator"* and it hardcoded the judge to
`claude-sonnet-4-6`.

The rationale left in the code was that the judge must not be the same model as the
optimizer. True — but it was solved with a different *checkpoint of the same vendor*.
Gemini is a different **vendor**: no shared weights, training data, or house style to
pattern-match its own output as "good".

**Restored:** Claude Opus 4.8 optimizes, Gemini critiques.

Verified working end to end: Gemini scored a candidate **7.0 and rejected it** with specific
feedback, Claude revised, Gemini **passed it at 8.0**. Real critique, real revision.

### Why it had been unusable at volume (and why it was probably swapped out)

Gemini's CLI latency scales **super-linearly** with prompt size — measured:

| prompt | latency |
|---|---|
| 5 KB | 7 s |
| 20 KB | 17 s |
| 50 KB | 142 s |

Optimizer candidates run 10–35 KB, and the evaluator timeout was **300 s**. A real task
(task 1) lost **3/3 iterations to timeouts** and recorded a score of **0.0** — which reads
as "Claude produced garbage" but actually means "the judge never ran."

**Fixed:**
- evaluator timeout 300 s → **900 s**
- an unparseable/timed-out evaluation now **retries** instead of recording a phantom 0.0
  (proven: 2 timeouts → recovers to a real 9.0)
- the proposal file is now a **full review transcript** — Gemini's complete critique and
  Claude's actual work at every iteration. It previously truncated feedback to **200
  characters** and never recorded the candidate, so it could not answer "how good is the
  optimizer doing?" — the one question it existed to answer.

## 3. Does the desktop app work? Yes — mostly.

Every RPC the desktop UI can call was executed against the real `four_in_a_line` solution
by driving the sidecar's own NDJSON loop.

- **Read paths: 46/53 pass.** The 7 "failures" are correct *not-found* responses to
  deliberately fake ids ("proposal not found: none"). Effectively the whole read surface works.
- **Write paths: verified** — `operator.set`, `goals.create`, `backlog.submit`,
  `knowledge.add`, `analyze.run` all land real records.
### The full five-phase lean loop is verified end to end

Two proposals were created, one **approved** and one **rejected with feedback**, straight
through the sidecar — the same path the desktop UI uses. Audit log before: **3 events**.
After: **6**.

```
audit by_action_type:
  ANALYSIS_PROPOSAL: 3      <- SURFACE + PROPOSE
  PROPOSAL_APPROVED: 1      <- DECIDE (approve)
  PROPOSAL_REJECTED: 1      <- DECIDE (reject)
  FEEDBACK_LEARNING: 1      <- COMPOUND (Phase 5)
```

And the rejection feedback was **retrievable from vector memory** afterwards — the
correction genuinely fed the compounding-memory loop.

**SURFACE → CONTEXTUALIZE → PROPOSE → DECIDE → COMPOUND all work.** The approval gate, which
SOUL.md calls "the product," holds: proposals are queued, decisions are signed and recorded,
and rejections teach.

> **Third false critic finding.** Gemini's `approvals` review alleges *"silent audit-log
> failures"* on rejection. It is wrong: rejection wrote both `PROPOSAL_REJECTED` and
> `FEEDBACK_LEARNING`. This is exactly why the reviews must be triaged, not executed.

The belief that "SAGE doesn't work" is not supported by evidence. What is true is that the
*surface area* is enormous (24 features) and thinly finished — most features are "partly"
usable, with rough empty states and error handling.

## 3b. What "drift" actually looks like on disk

The audit found the mechanism behind the sense that work never lands. `git status` shows
**four sidecar handlers that were never committed**:

```
?? sage-desktop/sidecar/handlers/health.py
?? sage-desktop/sidecar/handlers/knowledgesync.py
?? sage-desktop/sidecar/handlers/activity.py
?? sage-desktop/sidecar/handlers/agentrun.py
?? sage-desktop/handlers_knowledgesync_placeholder
```

These are half-finished features abandoned mid-write. It is exactly why `_PROBE_POOL` was
undefined — `health.py` was left incomplete and never committed, so nothing ever compiled it.

Two more, still open:

| Issue | Detail | Status |
|---|---|---|
| **Console / Logs page is broken** | `Console.tsx` imports `useLogTail` from `useLogs.ts`, which imports `tailLogs` and `LogEntry` — **neither exists** in the API client. The page would crash at runtime. `handlers/logs.py` exists but **was never registered** in the dispatcher. | **Open** — a genuinely unfinished feature, not a small bug |
| **`tsc` fails: 5 errors in 3 files** | `KnowledgeSync.tsx`, `useKnowledgeSync.ts`, `useLogs.ts` reference client exports that don't exist. `vite build` still passes (esbuild strips types), so this is invisible until CI typechecks or the page renders. | **Open** |

Fixed during this audit (Gemini found it; I verified it against the source before fixing):

| Issue | Detail | Status |
|---|---|---|
| **Workflows approval gate never updated** | `activeRun` was `resume.data ?? run.data` — set only when a mutation resolved. "Refresh status" updated a separate badge, so the HITL approval panel (gated on `activeRun.status === "awaiting_approval"`) never appeared when a run reached the gate, and never cleared when it finished. | **Fixed** — refreshed status is folded back into `activeRun`, guarded on `run_id`. 349 vitest tests pass. |

The pattern is not a model that cannot code. It is **features started, left mid-air, and never
verified** — which is what an unbounded scope with no working definition of "done" produces.

## 4. Feature-by-feature review — `docs/reviews/`

One file per feature (24), each reviewed by Gemini against **live runtime evidence**
(the actual RPC responses), plus the handler and UI source. See
[`docs/reviews/_INDEX.md`](./reviews/_INDEX.md).

Critic's summary: **0 yes, 21 partly, 3 no. Mean 4.3/10.**

### ⚠️ Do not feed these to the optimizer unverified

Two findings were checked by hand, and **both were false — for different reasons.** They are
worth understanding, because between them they explain a great deal.

**1. The critic hallucinates — consistently, which is worse than randomly.** Gemini's
`llm.md` review scores the feature **1/10 "completely broken"** on two *High Severity*
findings:

> *"Line 51 of `llm.py` contains corrupt, un-compilable text: `@.platformio\python3\Lib\__pycache__\dataclasses.cpython-311.pyc`"* … *"the `@dataclass` decorator was overwritten"*

**Both are fabricated.** Line 51 is `gw = _require_gateway()`. The `@dataclass` decorator is
present on line 59. The file compiles (`py_compile`: clean) and served a live RPC in the same
run — which Gemini explains away by speculating the runtime "must be using a cached module."

It invented **the same fake detail across two independent runs**. A one-off hallucination is
noise; a reproducible one reads like a real bug report and will survive review.

**2. A damaged instrument manufactures defects.** `audit.md` scored **2/10, "completely
unusable,"** because the JSON *"truncates mid-field."* It did — **but the audit harness
truncated it**, not SAGE. Gemini reasoned correctly from evidence *I* had broken. After
labelling truncation explicitly, the finding **vanished entirely** and `audit` re-scored to
**5/10, "partly usable"** with a substantive real finding instead.

**3. But the critic is not useless — it found a real bug.** Gemini's `workflow` review
identified a genuine state-desynchronisation defect in `Workflows.tsx` (see §3b). Verified
against the source, then fixed.

That is the whole lesson in one place: **the critic is worth running and cannot be trusted.
Verify every finding before acting on it.**

*(An earlier draft of this report claimed "6 of 24 reviews make a disproven syntax-error
claim," from a substring match. That was itself an unverified generalisation — four of the six
merely mention syntax errors in benign contexts. It has been corrected. The irony is the
point: this is exactly the failure mode the section warns about, and it caught the author of
the section.)*

This is not a footnote — it is likely a **root cause of the "months of drift."** An optimizer
fed unverified critic output will spend hours fixing bugs that do not exist, and the work
will look like failure. Every finding is a **hypothesis to verify**, not a work order.

The scores are the critic's impression. **They are not a measure of how well Claude — or any
model — performed.** Nothing in these reviews was produced by an optimizer.

## 5. On the model question

The 0.0 on task 1 did **not** mean Claude produced garbage. Claude produced a 34 KB typed-schema
proposal; the *judge* fell over and never scored it. Had that number reached a summary table
without the transcript, it would have read as "Claude failed."

Before switching vendors, the honest test is a **bake-off** — SAGE is multi-provider by design.
Run the same tasks with a Claude, GPT, and Qwen optimizer against the same rubric and the same
independent Gemini critic, and compare transcripts. That settles the question with data
instead of argument. Nothing in this audit's evidence points to model capability as the
bottleneck; it points to **unverified instruments and unbounded scope**.
