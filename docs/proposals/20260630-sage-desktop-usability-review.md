# sage-desktop review report

## VERDICT
NO — not usable today as the only interface, blocked on four independent counts. (1) It does not launch out of the box: two compounding Rust bugs — the default SAGE_ROOT derivation is wrong (current_exe().parent().parent() resolves to src-tauri/target in dev and ...\\Programs when packaged, so the sidecar never spawns; lib.rs:29-32 + sidecar.rs:63-68), AND on spawn failure handle.manage() runs only on the Ok branch (lib.rs:64-72) so the Sidecar state is never registered and every command rejects with unmanaged-state. Fixing only one still ships a dead UI; both must be fixed together. (2) It cannot be installed: tauri.conf.json has no externalBin/resources, so make desktop-build bundles neither Python nor the sidecar, and the packaged SAGE_ROOT is wrong anyway; the two 295 MB bin/ exes are orphaned (never referenced). (3) The core product loop is half-open: the dispatcher registers no operator trigger to make an agent analyze a signal and PROPOSE (no analyze/agent.run/chat/developer/code/swe/mr/planner methods; app.py:69-117). The only two work-creating methods, builds.start and onboarding.generate, do NOT write to ProposalStore (builds.approve is its own state machine), so the headline Approvals inbox has no operator-reachable feed — operators can DECIDE but cannot generate anything to decide on. (4) Persona-blocker: the interface built explicitly for compliance teams cannot surface compliance status (no compliance.* RPC for domains/checklist/gap-assessment); note this is the assessment tooling — the legally load-bearing audit RECORD itself IS present (audit.list/get_by_trace/stats), so it is a conformance-assessment gap, not a missing record. Once (1)-(3) are fixed it becomes operable for the DECIDE/COMPOUND half (approvals, audit, knowledge, constitution, collective, YAML, builds, onboarding, backlog triage, LLM/solution switch); (4) gates the stated regulated-industry persona specifically.

## TEST SUMMARY
Raw counts are all green: 407/407 passed across the three make test-desktop layers — sidecar pytest 211/0 (72s, Python 3.12.9/pytest 9.0.3), Rust cargo 20/0 (40s), React vitest 176/176 (16s); all toolchains installed and buildable, no ports/services/admin needed. But the green is structurally misleading, and that is the honest pass/fail picture: the launch-blocking bugs shipped precisely because they live in the gap between what make test-desktop covers and what would exercise a real launch. The Rust suite runs cargo test --lib --no-default-features, compiling with the desktop feature OFF, so lib.rs desktop_app and commands/* are never compiled or run — the SAGE_ROOT path bug and the manage-only-on-Ok state bug are entirely untested. The sidecar queue tests pass only because test_queue.py uses a FakeQueue that fabricates a _config attribute the real TaskQueue lacks, masking the silent-wrong parallel-config readout. The one suite that drives a real launch end to end (make test-desktop-e2e, requires tauri-driver) is NOT part of make test-desktop and was not run. So the suites prove the pure Rust modules and the sidecar handlers behave correctly in isolation; they do NOT prove the app launches, installs, or operates as a whole. Minor harmless noise: desktop-gui.md cites stale-low counts (176 sidecar / 152 vitest vs actual 211 / 176) and React Router v7 future-flag warnings.

## BLOCKING GAPS
- App does not launch out of the box: wrong default SAGE_ROOT derivation (sidecar never spawns; lib.rs:29-32 + sidecar.rs:63-68) AND state managed only on the Ok branch so spawn failure leaves every command rejecting (lib.rs:64-72) — two compounding bugs, both must be fixed or the window opens to a dead UI.
- App cannot be installed/distributed: tauri.conf.json has no externalBin/resources, so the installer bundles no Python and no sidecar; packaged SAGE_ROOT (2-up) is also wrong, so even a present interpreter would not resolve from src.* — packaged app is broken on two independent counts. (590 MB of orphaned bin/ exes are dead weight.)
- No operator trigger for the PROPOSE half of the loop: no analyze/agent.run/chat/developer/code RPC, and the only work-creating methods (builds.start, onboarding.generate) do not write to ProposalStore — so the Approvals inbox, the Phase-1 headline feature, has nothing a desktop-only operator can put in it. The product loop is half-open.
- Compliance assessment tooling is unreachable from the interface built for compliance teams: no compliance.* RPC (domains/flags/checklist/gap-assessment, IEC 62304 / 21 CFR Part 11 matrices). The audit RECORD is present; what is missing is conformance ASSESSMENT — a persona-blocker for the stated regulated-industry use case.

### sage-desktop Python sidecar (stdin/stdout NDJSON JSON-RPC 2.0, no HTTP)
- **[broken/major]** queue.get_status silently reports wrong parallel config. The handler reads `getattr(_queue, "_config", None)` and falls back to parallel_enabled=False/max_workers=0 when absent. The object wired at app.py:216 is `get_task_queue(name)` -> a bare `TaskQueue` whose __init__ (queue_manager.py:282-289) defines NO `_config`. The live parallel config lives on `parallel_runner.config`, which is exactly what the FastAPI equivalent reads (api.py:1716-1717). So against the real framework this RPC ALWAYS returns parallel_enabled=false, max_workers=0, regardless of actual runtime config. The per-status counts (pending/in_progress/done/failed/blocked) are correct; only the documented 'parallel config' fields are wrong. The unit test hides this with a FakeQueue that fabricates `_config` (test_queue.py:9-12).
    EVIDENCE: sage-desktop/sidecar/handlers/queue.py:32-36; src/core/queue_manager.py:282-289 (no _config) vs :954 (config on ParallelTaskRunner); src/interface/api.py:1716-1717; sage-desktop/sidecar/tests/test_queue.py:9-12
- **[broken/minor]** parse_request drops a valid request id on a params-shape error, breaking Rust oneshot correlation. id is validated first, then `if not isinstance(params, dict): raise RpcError(RPC_INVALID_PARAMS, ...)` is raised BEFORE the Request is constructed/returned. In the app.py loop `req` is therefore still None in the except handler, so the error frame is emitted with id=null (app.py:268-277) even though the incoming frame carried a real id. The Rust RpcClient correlates responses to pending requests by id, so a null-id error cannot be matched and that request would hang until timeout. Trigger: a frame with a valid id and `params` that is a list/number/string. Low real-world likelihood (the Rust client always sends object params) but it is a concrete framing defect on the error path.
    EVIDENCE: sage-desktop/sidecar/rpc.py:70-78; sage-desktop/sidecar/app.py:268-277
- **[broken/minor]** queue.* is not solution-isolated. The sidecar passes solution_name to get_task_queue (app.py:216), implying per-solution scoping, but get_task_queue constructs `TaskQueue()` with no db_path (queue_manager.py:1208), so every instance uses the module-global `_DB_PATH` = <repo>/data/audit_log.db (queue_manager.py:41-44). Proposals and audit ARE wired per-solution under solution_path/.sage/ (app.py:172,181), so queue.get_status/list_tasks surface a framework-global task queue under a solution-scoped desktop UI, contradicting the doc's 'one sidecar per solution / data isolation' claim. It mirrors FastAPI's existing global-queue default (api.py:1710 module singleton), so it is not a sidecar-introduced regression, but the per-name registry creates a false appearance of isolation.
    EVIDENCE: src/core/queue_manager.py:41-44, :1199-1209; sage-desktop/sidecar/app.py:213-218 vs :169-194

### Rust/Tauri shell (sage-desktop/src-tauri)
- **[broken/blocker]** Default SAGE_ROOT derivation is wrong, so the sidecar never spawns out of the box. default_sidecar_config() falls back to current_exe().parent().parent() (2 levels up). In dev the exe is src-tauri/target/debug/sage-desktop.exe, so SAGE_ROOT resolves to src-tauri/target and sidecar_dir = src-tauri/target/sage-desktop/sidecar (nonexistent). Sidecar::spawn then sets current_dir to that dir's nonexistent parent, so Command::spawn fails on Windows -> DesktopError::SidecarDown. make desktop-dev does not export SAGE_ROOT, so the documented default launch path is non-functional unless a user manually sets SAGE_ROOT. desktop-gui.md:184 claims this 'just works'; it does not.
    EVIDENCE: src-tauri/src/lib.rs:26-34 (parent().and_then(|p| p.parent())); src-tauri/src/sidecar.rs:63-68 (current_dir on sidecar_dir.parent()); Makefile:187-189 (desktop-dev sets no env)
- **[broken/blocker]** On sidecar spawn failure the app launches but every command errors. lib.rs calls handle.manage(RwLock::new(sidecar)) ONLY on the Ok branch; on Err it just logs. So the State<RwLock<Sidecar>> is never registered and every #[tauri::command] that takes that state rejects with Tauri's unmanaged-state error. Combined with the path bug above, the window opens but the whole UI is dead. There is also a happy-path race: spawn runs in a detached async task, so commands are invocable before the state is registered (App.tsx's 'retry lightly' comment papers over exactly this).
    EVIDENCE: src-tauri/src/lib.rs:60-73 (manage only on Ok, error only logged); src-tauri/src/commands/solutions.rs:14-22 (State<'_, RwLock<Sidecar>> usage); src/App.tsx:23

    EVIDENCE: src-tauri/tauri.conf.json:28-38 (bundle block: active/targets/icon only, no externalBin/resources); .claude/docs/interfaces/desktop-gui.md:75-79
